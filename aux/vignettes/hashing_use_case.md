# Vignette: Reliable Agentic Bioinfo Pipelines (Hashing vs. Targets)

This vignette demonstrates how **HydraR** handles long-running, non-deterministic bioinformatics workflows using its native **State Hashing and Restart engine**.

## Scenario: The "Friday Night Crash"
A researcher initiates a complex agentic workflow to:
1. **Node A**: Perform Differential Expression (DE) on a 10,000-sample proteomics dataset using `limma`. (Time: 45 min)
2. **Node B**: Perform Functional Enrichment using `clusterProfiler`. (Time: 20 min)
3. **Node C**: Generate a high-resolution, annotated `ComplexHeatmap`. (Time: 10 min)

**The Problem**: At step 3, the R session crashes due to a memory overflow in the graphics device.

## The HydraR Solution (Native Hashing)
Unlike `targets`, which requires a static file-based graph, HydraR hashes the **State + Logic + Driver** at every node transition.

### 1. The Initial Run
```r
library(HydraR)

# Define a persistent DAG
dag <- dag_create(persistent_path = "research_cache.duckdb") |>
  dag_add_logic_node("de_analysis", limma_logic_fn) |>
  dag_add_logic_node("enrichment", cluster_profiler_fn) |>
  dag_add_logic_node("viz", heatmap_viz_fn)

# Run starts...
dag$run() 
# [Logic: de_analysis] DONE. Hash: a1b2c3...
# [Logic: enrichment]  DONE. Hash: d4e5f6...
# [Logic: viz]         FAILED (Memory Error)
```

### 2. The Resilient Restart
Because HydraR stored the hashes of `de_analysis` and `enrichment` in its DuckDB registry, the researcher can simply rerun the command:

```r
dag$run(restart = TRUE)
# [Logic: de_analysis] SKIPPED (Hash match: a1b2c3...)
# [Logic: enrichment]  SKIPPED (Hash match: d4e5f6...)
# [Logic: viz]         RESUMING... DONE.
```

## Why this replaces `targets` for Agents
1. **Agent Awareness**: Agents often generate dynamic branching (e.g., "if p-value is high, try another contrast"). `targets` cannot easily manage the combinatorial explosion of branches generated by an LLM at runtime.
2. **Zero-Configuration**: In HydraR, hashing is "on by default" for all `idempotent = TRUE` nodes. No `_targets.R` file is required.
3. **Internal Transparency**: The hashes are stored in the same DuckDB as the **Message Log**, allowing for a unified audit trail of *what* was run and *why* it was skipped.
